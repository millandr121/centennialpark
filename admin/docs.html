System Objective

Build and maintain a fully operational park management system for Eileen Scott Centennial Park in Bamfield, BC. The system must remain simple to operate even as features grow — automation is good, complexity visible to the manager is bad.

Priorities

  1. Reservation system — online booking form (public-facing) + admin management panel (password-gated)
  2. Email notifications — instant emails to park and guest on every booking/contact; daily digest as safety backup
  3. Financial tracking — all-in amounts, GST extracted automatically, misc income ledger, period income/GST reports
  4. Usability — automated where possible; 100% editable and readable by the manager
  5. Reliability — D1 database as durable backup; graceful degradation when any service is unavailable
Guiding principle: if something breaks, the manager is never left without a fallback. Forms show "email us directly", the daily digest catches missed emails, and the database always has a copy.

How deploys work

Every push to the main branch on GitHub triggers an automatic Cloudflare Pages build. No manual steps needed. Changes are live within ~1 minute.

Files at the repo root are publicly accessible by URL — but none of them contain secrets. All credentials (passwords, API keys) live in Cloudflare environment variables, never in files.

Admin panel security

The /admin/ path is protected by HTTP Basic Auth via functions/admin/_middleware.js. The password is stored in the ADMIN_PASSWORD Cloudflare environment variable. All /admin/ pages are marked noindex so search engines don't index them.

GST calculation

All amounts are all-in (GST already included, not added on top). GST is extracted as:

GST = amount × 5 / 105

Example: guest pays $105 → $5 is GST, $100 is the base. The manager enters the one total the guest pays; the system handles the rest. GST-exempt items (donations, etc.) can be toggled off per line.

Admin Panel How-To

Tabs overview

Marking a reservation as paid

  1. Find the reservation in the Reservations tab
  2. Click the Unpaid badge — a confirmation dialog asks "Mark #N as PAID?"
  3. Confirm → a second dialog asks "Email the guest to notify them it is now paid?"
  4. Choose Yes or No — the reservation updates immediately

Editing a reservation

  1. Click the pencil icon on any reservation row
  2. Edit dates, amounts, notes, status
  3. Amount Paid — enter the all-in total the guest actually paid. GST auto-calculates.
  4. Check GST Exempt if applicable (donations, etc.) — GST zeros out
  5. Click Save

Recording misc income

  1. Go to Add → Misc Item
  2. Choose category (launch, shower, parking, donation, etc.)
  3. Enter date, description, amount
  4. GST auto-calculates; toggle exempt if needed
  5. Click Add Item

Misc items appear in the Income & GST report alongside reservation income.

Running income reports

  1. Go to Reports → Income & GST
  2. Select a preset (This Month, Last Month, This Year, etc.) or enter custom dates
  3. Click Generate
  4. Print — opens browser print dialog (report only, no nav/controls)
  5. Export CSV — downloads per-transaction detail file

Running booking reports

  1. Go to Reports → Bookings
  2. Filter by date range, type (camping/moorage/launch/parking), site, status (paid/unpaid/requested/all)
  3. Click Generate
  4. Click Print

Wiping the database

Found in Settings → Danger Zone. Scopes:

  • Reservations only — clears bookings; reference numbers restart from 1
  • Submissions only — clears form submissions
  • Misc Items only — clears the income ledger
  • Wipe ALL — clears everything; all IDs reset to 1
Wipes are permanent and cannot be undone. Use at season start or for testing only.

Email Service (Resend)

How it works

When a guest submits the booking or contact form, the Cloudflare Function calls the Resend API to instantly email the park Gmail. The same API sends the guest a confirmation. All sends are also saved to the D1 database first.

Free tier limits

LimitValueStatus
Emails per month3,000Very comfortable
Rate limit5 requests/secondNot a concern
Daily limitNoneOK

At typical campground volume (~10 bookings/day peak, confirmations + park notifications + daily digest), estimated monthly usage is 200–400 emails — well under the 3,000 limit.

What happens when email fails

The form submission is always saved to D1 first. If Resend fails (quota, outage, misconfigured key):

  • The form still shows success to the guest
  • Data is safe in the database
  • The daily digest Worker emails a recap the next morning

Resend logs are visible in the Resend dashboard. There is no "log overflow" that would block sending — logs are just history.

Sending domain

The RESEND_FROM address must use a domain verified in Resend. Until a domain is verified, Resend only allows sending to the Resend account owner's email address (test mode). To send to guests, verify your domain via DNS records in the Resend dashboard.

Database (Cloudflare D1)

What it is

Cloudflare D1 is a serverless SQLite database hosted at Cloudflare's edge. It's accessed only by the Cloudflare Functions via the DB binding — not publicly accessible. Data is durable (replicated) and survives indefinitely.

Tables

TableContents
reservationsAll bookings — manual and form-submitted
booking_submissionsRaw public booking form submissions (pre-review)
contact_submissionsRaw contact form submissions
misc_itemsNon-booking income entries (launch, shower, parking, donations)
sitesCampsite and moorage definitions
sqlite_sequenceInternal SQLite AUTOINCREMENT counters — cleared on database wipe

Running a manual query

Cloudflare Dashboard → D1 → centennialpark → Console. Example:

SELECT created_at, first_name, last_name, email FROM reservations ORDER BY created_at DESC LIMIT 20;

Schema migrations

New columns or tables must be added manually in the D1 console. Migration files:

  • schema.sql — original setup (run once)
  • schema-items.sql — adds gst_exempt, paid_at, creates misc_items
The API functions are backward-compatible — they check whether columns exist before using them. So the site won't break before a migration is run, it just won't use the new features yet.

Troubleshooting

Forms show "please email us directly"

Both Resend AND D1 failed at the same time. Check: Is the DB binding set in Cloudflare? Is RESEND_API_KEY set? Was a redeploy triggered after adding env vars?

Emails from forms not arriving

Check the Resend dashboard for send logs. Verify the sending domain is verified. Check if monthly quota (3,000) is exhausted. Note: the daily digest Worker still runs as a backup even if instant emails fail.

Admin panel login fails

The ADMIN_PASSWORD env var is missing or was typed incorrectly. Set it in Cloudflare dashboard and redeploy.

Reference numbers not resetting after a database wipe

Old wipe code didn't clear sqlite_sequence. Ensure the version of cleanup.js from PR #6 or later is deployed (checks for sqlite_sequence clearing on wipe).

New features (GST exempt, paid date, misc items) not working

Run schema-items.sql in the D1 console: Cloudflare Dashboard → D1 → centennialpark → Console → paste the full file contents and run.

Cloudflare build failing

Check Cloudflare Dashboard → Workers & Pages → centennialpark → Deployments for error logs. Most common cause: syntax error in a function file. Fix in GitHub and push again.

Site not updating after a GitHub push

Check the Cloudflare Pages deployment list. Builds usually complete within 60 seconds. If the build is stuck, try a manual retry from the Cloudflare dashboard.

Daily digest not arriving

Check the digest Worker's logs in Cloudflare Dashboard → Workers → (digest worker name). Common causes: RESEND_API_KEY not set on the Worker, database binding not set, or the cron trigger wasn't deployed. To test manually, visit the Worker's URL in a browser.
